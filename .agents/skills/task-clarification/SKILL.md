# Task Clarification
Контекст задачи: $ARGUMENTS

## Цель
Зафиксировать понимание задачи и получить явное подтверждение пользователя до любых изменений.

## Жесткие ограничения
1. Не изменяй файлы и не запускай мутационные команды в этой фазе.
2. Не переходи к реализации, пока пользователь явно не подтвердил контракт.
3. Если есть минимум две разумные трактовки, обязательно задай уточняющие вопросы.

## Порядок работы
1. Сформируй черновик контракта задачи.
2. Если не хватает данных, задай 1-5 точных вопросов по критическим пробелам.
3. Верни пользователю финальный контракт и Task Card.
4. Запроси явное подтверждение ("ОК", "подтверждаю", "делай").
5. Только после подтверждения подготовь краткий план этапов исполнения (без самой реализации в этом skill).

## Формат контракта
- Цель
- Границы изменений (что можно/нельзя менять)
- Ограничения и запреты
- Критерии готовности (Definition of Done)
- Минимальная проверка/валидация
- Риски и спорные места

## Формат Task Card
- Тип задачи: bugfix | refactor | feature | test
- Цель
- Границы изменений
- Запреты
- Критерии готовности
- Минимальная проверка

## Правила уточняющих вопросов
- Вопросы должны быть конкретными и проверяемыми.
- Не задавай вопросы, ответы на которые уже явно даны пользователем.
- Приоритизируй вопросы, влияющие на архитектуру, область изменений и критерии приемки.

## Примеры запросов, которые должны триггерить этот skill
- "Помоги сделать фичу, пока не понимаю как лучше"
- "Надо поправить это, но не уверен в границах изменений"
- "Сформируй четкое ТЗ перед реализацией"
- "Проверь, правильно ли ты понял задачу до начала кода"
