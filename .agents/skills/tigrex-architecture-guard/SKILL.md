# TIGREX Architecture Guard
Контекст задачи: $ARGUMENTS

## Цель
До начала реализации превратить идею в непротиворечивый архитектурный контракт, совместимый с `TIGREX_ARCH_README.md`.

## Когда использовать
- Перед feature/bugfix/refactor, где затрагивается архитектура.
- Когда есть риск противоречий в терминах `Action`, `Agent`, `Feature`, `EventBus`.
- Когда нужно решить, нужен orchestration слой или достаточно агента.

## Обязательные источники
1. `TIGREX_ARCH_README.md` (каноничные правила проекта).
2. Запрос пользователя и его ограничения в текущем диалоге.

## Жесткие ограничения
1. Не вносить изменения в код до фиксации архитектурного контракта задачи.
2. Не предлагать решения, нарушающие Core v1.0 и направленность зависимостей слоев.
3. Не допускать прямые `Agent -> Agent` вызовы в целевой архитектуре.
4. Не размывать `Action-first` модель строковыми/непроверяемыми контрактами.

## Порядок работы
1. Классифицируй задачу: single-area или cross-area сценарий.
2. Выдели сущности: какие `Action`, какие `Agent`, какая `Feature`, какие сервисные интерфейсы.
3. Прогони проверку на инварианты из `TIGREX_ARCH_README.md`.
4. Реши путь выполнения:
   - Agent-only (+ middleware), если сценарий локальный.
   - Orchestration слой, если сценарий межагентный/длинный/с компенсациями.
5. Сформируй “Architecture Task Card”.
6. Если есть двусмысленность, задай уточняющие вопросы и получи подтверждение.
7. Только после подтверждения переходи к плану реализации.

## Формат Architecture Task Card
- Тип задачи: feature | bugfix | refactor | test
- Цель изменения
- Затронутые `Action`
- Ответственные `Agent`
- Граница `Feature`
- Нужен ли orchestration: yes/no + причина
- События EventBus (если есть): какие и зачем
- Запрещенные решения для этой задачи
- Критерии готовности
- Минимальная валидация (сценарии/проверки)

## Red Flag проверка
Перед подтверждением проверь:
- Не появился ли god-agent/god-module.
- Не размазан ли business flow по eventbus.
- Нет ли дублирования ролей между feature/agent/service.
- Не появились ли правила, противоречащие `TIGREX_ARCH_README.md`.

## Выходной формат
Возвращай пользователю:
1. Краткий вердикт по архитектурной совместимости.
2. Architecture Task Card.
3. Список точек риска (если есть) и как их закрыть.
4. Явный запрос подтверждения перед реализацией.

## Примеры запросов-триггеров
- "Проверь архитектурно мою идею перед кодом"
- "Нужен ли тут оркестратор или достаточно агента?"
- "Хочу добавить фичу, не нарушив мои правила"
- "Поймай противоречия в моем дизайне до реализации"
