# C++ Refactor
Контекст задачи: $ARGUMENTS

## Цель
Сделать безопасный refactor с предсказуемым результатом и минимальным риском архитектурной деградации.

## Жесткие ограничения
1. До реализации верни Refactor Contract и запроси явное подтверждение пользователя.
2. Не меняй ядро/библиотеки, которые пользователь пометил как неизменяемые.
3. Не используй макросы.
4. Не добавляй хардкод абсолютных путей.
5. Не меняй внешнее поведение без отдельного согласования (иначе это уже feature/bugfix, а не refactor).

## Архитектурные требования к refactor
- Уменьшай связность между модулями.
- Держи модули компактными и с единой ответственностью.
- Старайся избегать «больших универсальных» файлов с разнородной логикой.
- Низкоуровневые модули должны быть самодостаточными.
- Для взаимодействия модулей одного уровня предпочитай событийную/горизонтальную связность.
- Удаляй лишний и дублирующий код, если это не ломает контракт поведения.

## Refactor Contract (обязателен до начала работ)
- Цель refactor
- Что остается неизменным по поведению
- Границы изменений (какие файлы/модули можно менять)
- Что запрещено менять
- Ожидаемый структурный результат (какая декомпозиция/упрощение)
- Риски
- Минимальная валидация после изменений

## Порядок работы (чекпоинты)
1. C1 - Контракт и согласование
- Верни Refactor Contract.
- Если задача неоднозначна, задай уточняющие вопросы.
- Дождись явного подтверждения.

2. C2 - План
- Дай краткий план изменений по модулям.
- Покажи, как изменится связность и границы ответственности.

3. C3 - Реализация
- Выполни минимально необходимые изменения.
- Не расширяй scope без повторного согласования.

4. C4 - Валидация
- Выполни сборку через `build.ps1` (если он используется проектом).
- Если есть релевантные проектные тестовые скрипты, прогони минимально достаточный профиль.

5. C5 - Отчет
- Что изменено структурно.
- Что не менялось по контракту.
- Чем подтверждена эквивалентность поведения.
- Какие риски/долги остались.

## Правила эскалации
- Если в процессе выяснилось, что без изменения поведения цель недостижима - остановись и запроси решение пользователя:
  - A: пересмотреть контракт и разрешить поведенческие изменения;
  - B: сузить refactor до безопасной части.

## Формат короткого отчета по refactor
1. Контракт (подтвержден/нет).
2. Затронутые модули.
3. Ключевые структурные изменения.
4. Результаты проверки.
5. Остаточные риски.

## Примеры запросов, которые должны триггерить этот skill
- "Сделай рефакторинг этого модуля без изменения поведения"
- "Разбей большой файл на более маленькие компоненты"
- "Уменьши связность между транспортом и бизнес-логикой"
- "Убери дубли и упрости структуру"
- "Переделай эту логику"
