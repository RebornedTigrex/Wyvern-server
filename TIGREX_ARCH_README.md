# TIGREX Architecture README (Prototype Platform) - v3
Документ фиксирует целевую упрощённую архитектуру прототипа универсальной сервер-платформы.
Цель платформы: новый сервер собирается как конструктор из плагинов, где разработчик в основном пишет фронтенд и прикладные API-обработчики.
Если разделы конфликтуют, всегда побеждает `Architecture Contract (MUST)`.

## 1) Цель и рамки текущей версии
- Платформа ориентирована на `HTTP API + выдачу страниц + выдачу статических/прикладных файлов` как базовый MVP.
- Архитектура строится по идеологии plugin-first: функциональность добавляется подключением модулей, а не переписыванием ядра.
- Базовая реализация - `C++-first`, при этом внешние API-контракты должны оставаться удобными для веб-разработки.
- Этапы миграции в этом документе не ведутся: устаревшие или ненужные элементы вырезаются, если они не нужны целевой архитектуре.
- Документ единый и самодостаточный: отдельный документ обязателен только для спорных архитектурных решений (Mini ADR).

## 2) Core принципы
- `Core v1.0` не изменяется.
- Никакого хардкода путей/окружения: только конфиг и относительные/гарантированные пути.
- Модули маленькие, с одной зоной ответственности.
- Минимальная связность и отсутствие модулей «всё-в-одном».
- Горизонтальные взаимодействия сущностей одного ранга - через контракты и события, без плотных прямых зависимостей.
- Избегать макросов, если задачу можно решить типами, интерфейсами и обычными функциями.
- Для аудио действует жёсткое правило: только `SFU mediasoup WebRTC`, без WebSocket-сигналинга аудио.
- По умолчанию не поддерживается legacy-совместимость, если она не согласована отдельно.

## 3) Architecture Contract (MUST)
### 3.1 Упрощённые уровни абстракции
- `L0 Core`: неизменяемая библиотека Core v1.0 и её контракты.
- `L1 Runtime`: инфраструктурное ядро платформы (lifecycle, router, plugin registry, `Wyvern.PluginRuntime`, config, observability, transport adapters).
- `L2 Plugins`: доменные и интеграционные плагины (pages, static, api, auth, files, jobs, media и т.д.).
- `L3 App Composition`: сборка конкретного сервера (какие плагины включены, какие роуты зарегистрированы, какие политики применены).

Разрешённое направление зависимостей:
- Только вниз: `L3 -> L2 -> L1 -> L0`.
- Обратные зависимости запрещены.
- Обход слоя запрещён (например, `L3 -> L1` в обход `L2` для бизнес-операции).

### 3.2 Plugin как строительный куб
Каждый плагин обязан иметь явный контракт:
- `pluginId`, `version`, `kind`;
- `dependencies` (декларируемые зависимости);
- `configSchema` (валидация входного конфига);
- `register(container, bus, router)`;
- `init()`, `start()`, `stop()`;
- `health()` и минимальный диагностический статус.

Правила:
- Нельзя использовать плагин до регистрации и прохождения lifecycle.
- Циклические зависимости между плагинами - ошибка запуска.
- Скрытая инициализация вне lifecycle запрещена.
- Плагин обязан явно объявлять, какие действия (actions) и события он поддерживает.

### 3.3 Action-first модель
`Action` - атомарное типизированное намерение (`CreateUser`, `ServePage`, `UploadFile`, `GenerateReport`).

Контракт команды:
`{ RouteObject, Agent, Action, Context, Meta }`

Где:
- `RouteObject` - целевой plugin/feature;
- `Agent` - исполнитель внутри plugin (или alias обработчика);
- `Action` - типизированное намерение;
- `Context` - DTO с полезной нагрузкой;
- `Meta` - технический контекст (traceId, auth, deadline, tenant и т.д.).

Правила:
- Любая state-changing операция выполняется только через `command -> feature(plugin) -> agent`.
- Router делает только резолв, структурную валидацию и делегирование.
- Router не исполняет бизнес-логику.
- Бизнес-валидация инвариантов выполняется в агенте/сервисе плагина.

### 3.4 Упрощённая роль Feature и Agent
- `Feature` в рамках этой версии = доменный plugin.
- `Agent` = исполнитель группы близких use-case внутри одного plugin.
- Прямые вызовы `Agent -> Agent` запрещены.
- Межплагинная координация выполняется командами, контрактами сервисов и событиями по правилам ниже.

### 3.5 Command и Event policy
Использовать `Command`, если:
- нужно изменить состояние;
- нужен конкретный исполнитель;
- нужен предсказуемый результат и контролируемая ошибка;
- нужна трассируемая цепочка выполнения.

Использовать `Event`, если:
- нужно уведомить о факте уже выполненного действия;
- подписчиков может быть `0..N`;
- реакция интеграционная/побочная.

Запрещено:
- запускать ключевой бизнес-flow только через eventbus;
- подменять команду событием, чтобы обойти валидацию;
- переносить ответственность исполнения из агента в подписчиков событий.

### 3.6 Политика orchestration
Orchestrator вводится только если одновременно выполняются условия:
- сценарий межплагинный;
- flow длинный (несколько шагов с retry/timeout/compensation);
- нужна явная трассировка и контроль шага.

Если сценарий локальный для одного plugin - orchestration не нужен.
Orchestrator координирует шаги, но не исполняет бизнес-операции вместо агентов.

### 3.7 `Wyvern.PluginRuntime` (MUST)
Назначение:
- Единый переиспользуемый runtime-слой для всех Wyvern-проектов.
- Обеспечивает полный lifecycle плагинов без изменений `Core v1.0`.
- Убирает необходимость писать новый загрузчик плагинов в каждом проекте.

Граница ответственности:
- Принадлежит `L1 Runtime`, а не `L0 Core`.
- Не содержит прикладной бизнес-логики.
- Работает только с контрактами плагинов, конфигом и инфраструктурными зависимостями.

Состав `Wyvern.PluginRuntime`:
- `ManifestLoader`: загрузка описаний плагинов из конфигурации приложения.
- `ManifestValidator`: проверка структуры манифеста, версии и обязательных полей.
- `DependencyGraphResolver`: построение графа зависимостей и проверка циклов.
- `LifecycleCoordinator`: состояния `discovered -> validated -> registered -> initialized -> started -> stopped|failed`.
- `RouterBinder`: публикация action-контрактов плагина в router после успешной регистрации.
- `HealthAggregator`: сбор health/readiness по плагинам.

Минимальный контракт манифеста:
- `pluginId`
- `version`
- `kind`
- `dependencies` (required/optional)
- `actions` (экспортируемые действия)
- `events` (публикуемые/подписываемые события)
- `configSchemaRef` или эквивалент типизированной схемы
- `entrypoint` (или compile-time registration token)

Политика отказов:
- Отсутствует required dependency -> запуск приложения блокируется до исправления.
- Обнаружен цикл зависимостей -> запуск блокируется.
- Невалидная схема конфигурации plugin -> plugin не стартует, причина логируется структурированно.
- Для optional dependency допускается режим degraded, если это явно объявлено в манифесте.

Интеграция в приложение:
- Приложение (L3) передаёт в runtime список включённых плагинов и их конфиги.
- Runtime валидирует, сортирует порядок запуска и выполняет lifecycle.
- Router принимает только actions, опубликованные через runtime после успешной валидации.

Неподдерживаемые сценарии:
- Скрытая регистрация plugin «мимо runtime».
- Прямая ручная инициализация plugin в app-коде в обход `LifecycleCoordinator`.
- Бизнес-правила в компонентах `Wyvern.PluginRuntime`.

## 4) Минимальный обязательный runtime для MVP
Обязательные блоки:
- `HTTP Gateway` (входящие запросы, middleware, auth hook, request context).
- `Page/Static Delivery` (выдача страниц и статических ресурсов по контракту).
- `Command Router` (маршрутизация в plugin/agent/action).
- `Plugin Registry + Lifecycle Manager`.
- `Wyvern.PluginRuntime` (manifest loader + validator + dependency resolver + lifecycle coordinator + router binder).
- `Configuration Manager` (схемы, валидация, источники конфига).
- `Observability Stack` (логирование, метрики, трейсинг, health/readiness).

Опциональные блоки (подключаемые плагинами):
- фоновые задачи и очереди;
- внешние хранилища и интеграции;
- real-time и медиа-модули.

## 5) Категории плагинов
### 5.1 Foundation plugins
Низкоуровневые или сервисные плагины, которые дают базовые возможности runtime (auth/session/cache/storage adapters и т.д.).

### 5.2 Feature plugins
Плагины прикладной области: страницы, API-домены, файлы, уведомления, кабинет пользователя и т.д.

### 5.3 Integration plugins
Плагины интеграции с внешними системами (DB, object storage, message brokers, внешние API).

Общее правило:
- Feature plugin не должен знать детали транспорта/драйверов.
- Детали инфраструктуры скрываются за интерфейсами сервисного уровня.

## 6) Модель разработки API (FastAPI-like опыт в C++)
Цель: новую фичу можно добавить без переписывания ядра.

Поток разработки:
1. Описать DTO запроса/ответа и схему валидации.
2. Добавить `Action`.
3. Реализовать обработчик в агенте соответствующего plugin.
4. Связать HTTP route -> command (`RouteObject/Agent/Action`).
5. Зарегистрировать plugin в lifecycle, соотвественно router будет брать данные опубликованные в lifecycle.
6. Добавить минимальные проверки (валидность DTO, happy-path, error-path).

Правила:
- Обработчики API не должны работать напрямую с transport-деталями.
- Любая сквозная политика (auth, rate limit, tracing) оформляется middleware/infra-слоем, а не копипастой в handlers.
- Нельзя делать API-контракты неявными строковыми соглашениями без DTO-схем.

## 7) Наблюдаемость (MUST)
### 7.1 Логи
Структурированные логи обязательны.
Минимальные поля:
- timestamp, level, service;
- pluginId, agent, action;
- traceId/requestId, sessionId (если есть), userId/tenantId (если применимо);
- outcome, errorCode, durationMs.

Все логи записываются в файл (Как будет написано в конфиге), логгирование вызывается обращением к функции, в которой устанавливаются сопутствующие параметры.

### 7.2 Метрики
Минимально обязательные метрики:
- RPS и active requests;
- latency (p50/p95/p99) по endpoint и action;
- error rate по endpoint/action/plugin;
- queue depth и task wait time (если есть очереди);
- active sessions/connections;
- память/CPU процесса.

### 7.3 Трейсинг
Минимальная цепочка трейсинга:
`HTTP ingress -> router -> plugin -> agent -> service -> external dependency`.

### 7.4 Health/Readiness
- `liveness`: процесс жив.
- `readiness`: процесс готов принимать трафик с учётом зависимостей.
- Каждый plugin должен уметь вернуть собственный health-state.

## 8) Целевые NFR для прототипа
Приоритеты: скорость, наблюдаемость, одновременная работа `200+` пользователей.

Базовые критерии приёмки прототипа:
- Платформа не теряет управляемость и трассируемость под нагрузкой `200+` одновременных пользователей.
- Ошибки диагностируются по логам и метрикам без ручного «угадывания» пути запроса.
- Нет деградации до нерабочего состояния при смешанной нагрузке (страницы + API + операции с файлами).

Рабочие целевые бюджеты (можно уточнять через замеры и ADR):
- p95 для типового read API в целевом профиле нагрузки - контролируемый и стабильный между релизами;
- error rate в штатной нагрузке - не выше согласованного SLO;
- деградация производительности после добавления нового plugin - измеряемая и ограниченная заранее установленным порогом.

Если бюджет/порог изменяется, это фиксируется через Mini ADR.

## 9) Lifecycle и bootstrap
Порядок запуска:
1. Старт app composition и загрузка конфига.
2. Инициализация runtime-ядра (`eventBus`, router, registry, observability).
3. Регистрация и init foundation plugins.
4. Регистрация и init feature/integration plugins.
5. Автоматическая регистрация routes/actions в router через lifecycle.
6. `start()` по графу зависимостей.

Порядок остановки:
- Обратный граф зависимостей.
- Drain активных запросов и задач.
- Явный `stop()` каждого plugin.

## 10) Правила структуры кода
Рекомендуемая структура (пример):
- `platform/runtime/*` - ядро платформы.
- `platform/runtime/plugin_runtime/*` - реализация `Wyvern.PluginRuntime`.
- `platform/contracts/*` - публичные контракты/DTO/интерфейсы.
- `plugins/<plugin-name>/*` - реализация конкретного plugin.
- `apps/<app-name>/*` - композиция конкретного сервера.

Ограничения:
- Один файл/класс = одна доминирующая ответственность.
- Никаких абсолютных путей в коде.
- Запрещены скрытые глобальные mutable-состояния без явного контракта.
- Дублирующий код подлежит выносу в контрактный слой или удалению.

## 11) Жёсткие запреты (red lines)
Без отдельного обоснования и Mini ADR запрещено:
- превращать eventbus в основной движок бизнес-исполнения;
- создавать god-plugin или god-agent;
- размещать бизнес-логику в router/middleware;
- добавлять обход lifecycle через «быстрые» скрытые инициализации;
- возвращать legacy-ветки «на всякий случай»;

## 12) Controlled Experiments
Режимы:
- `Stable` - текущий контракт и поставка.
- `Lab` - экспериментальные гипотезы.

Правила:
- 1 эксперимент = 1 проверяемая гипотеза.
- Обязательны метрика успеха, срок проверки и условие отката.
- Без измеримого выигрыша эксперимент не переносится в Stable.
- Крупный эксперимент и критичная поставка не смешиваются в одном изменении.

## 13) Mini ADR (обязателен для спорных решений)
Формат:
1. Проблема.
2. Решение.
3. Отклонённые альтернативы.
4. Какие разделы архитектурного контракта затронуты.
5. Как проверялось (сценарии, нагрузка, метрики).

## 14) Scenario Gate для архитектурных изменений
Любое архитектурное изменение проходит минимум такие сценарии:
1. `200+` одновременных пользователей запрашивают страницы и API.
2. Смешанная нагрузка: API + статические файлы.
3. Два параллельных загрузчика файлов + три тяжёлых операции в веб-интерфейсе.
4. Подключение новой feature-плагины без изменения `Core v1.0`.
5. Полная трассировка запроса от входа до исполнительного action.

Если для прохождения сценария нужны «особые исключения», решение не готово для Stable.

## 15) Before Code Checklist
Перед началом реализации должно быть явно определено:
- какой `Action` добавляется/изменяется;
- какой `plugin/agent` отвечает за исполнение;
- как выглядит route-to-action путь;
- какие метрики/логи/трейсы добавляются;
- нет ли нарушения зависимости слоёв и lifecycle;
- не требуется ли Mini ADR из-за спорного решения.

Если любой пункт не закрыт, сначала уточняется контракт, затем пишется код.

## 16) Architecture Task Card (шаблон)
- Тип задачи: `feature | bugfix | refactor | test`
- Цель изменения
- Границы изменений
- Запреты
- Затронутые `Action`
- Ответственные `plugin/agent`
- Нужен ли orchestration: `yes/no` + причина
- События EventBus: какие и зачем
- Критерии готовности
- Минимальная проверка (включая Scenario Gate)
